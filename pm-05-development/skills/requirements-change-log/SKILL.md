# 需求变更记录自动生成

## 核心原则

1. **变更必记录**——任何需求变更都必须有正式记录，口头变更不生效
2. **影响必评估**——每次变更必须评估对进度、质量、范围的影响
3. **审批必留痕**——变更必须经过审批，审批结果可追溯
4. **追溯链完整**——从变更源头到最终实现，链路不断裂

## 交互模式

🤖→👤 AI建议人类审批

## 输入

| 输入项 | 类型 | 必填 | 来源 | 说明 |
|--------|------|------|------|------|
| PRD文档（当前版本） | Markdown | ○ | output/pm-design/design-prd/PRD-{产品名}.md | 变更前的PRD |
| PRD文档（新版本） | Markdown | ○ | 用户提供 | 变更后的PRD |
| SRS文档 | Markdown | ○ | output/pm-design/requirements-srs/SRS-{产品名}.md | 受影响的需求规格 |
| 变更描述 | string | 是 | 用户提供 | 变更的内容描述 |
| 变更原因 | string | 是 | 用户提供 | 为什么要变更 |
| 变更发起人 | string | 是 | 用户提供 | 谁提出的变更 |
| 当前迭代/Sprint | string | ○ | 用户提供 | 当前所处的迭代周期 |

## 执行步骤

### Step 1: 变更识别与分类

基于变更描述和PRD对比，识别变更内容：

**变更分类**：

| 变更类型 | 说明 | 示例 |
|----------|------|------|
| 新增需求 | 原PRD中不存在的功能 | 新增社交分享功能 |
| 删除需求 | 原PRD中的功能被移除 | 取消离线模式 |
| 修改需求 | 原PRD中的功能被修改 | 支付方式从2种增加到4种 |
| 优先级调整 | 需求优先级变化 | P2提升为P0 |
| 范围调整 | 需求的适用范围变化 | 从仅支持国内扩展到海外 |

**变更编号规则**：
```
CHG-{YYYYMMDD}-{序号}
示例：CHG-20250315-001
```

### Step 2: 影响评估

评估变更对项目的多维度影响：

**影响评估矩阵**：

| 影响维度 | 评估内容 | 影响等级 |
|----------|---------|---------|
| 范围影响 | 新增/减少了多少功能点 | 高/中/低 |
| 进度影响 | 是否影响当前迭代交付 | 延期X天/无影响 |
| 质量影响 | 是否引入新的风险点 | 高/中/低 |
| 资源影响 | 是否需要额外人力 | +X人天/无影响 |
| 技术影响 | 是否影响技术架构 | 架构变更/接口变更/无影响 |
| 测试影响 | 回归测试范围 | 全量/增量/无 |
| 依赖影响 | 是否影响上下游需求 | 需求A/B受影响/无 |

**受影响需求清单**：

| 受影响需求ID | 需求名称 | 影响类型 | 影响描述 |
|-------------|---------|---------|---------|
| FR-AUTH-001 | 用户注册 | 修改 | 新增邮箱注册方式 |

**受影响下游文档**：

| 文档 | 影响描述 | 是否需要更新 |
|------|---------|------------|
| SRS | 接口需求变更 | 是 |
| API契约 | 新增注册接口参数 | 是 |
| 测试用例 | 新增邮箱注册测试 | 是 |
| 设计稿 | 注册页面UI变更 | 是 |

### Step 3: 变更决策建议

基于影响评估，给出变更决策建议：

**决策建议**：

| 决策选项 | 适用条件 | 说明 |
|----------|---------|------|
| ✅ 接受变更 | 影响可控，收益明确 | 纳入当前迭代或下个迭代 |
| ⏳ 延期变更 | 影响较大但非紧急 | 推迟到后续版本 |
| ❌ 拒绝变更 | 影响过大或与战略不符 | 记录但不在当前版本实施 |
| 🔄 替代方案 | 变更目标合理但方案有更优解 | 建议替代实现方式 |

**决策依据**：
- 变更的业务价值 vs 实施成本
- 对当前迭代交付的影响
- 对产品质量的影响
- 对团队士气的影响

## 决策规则

| 条件 | 决策 |
|------|------|
| 变更影响>5个需求 | 标记为"重大变更"，建议升级审批 |
| 变更导致迭代延期>3天 | 标记为"高风险变更"，建议项目经理审批 |
| 同一需求3次以上变更 | 标记为"需求不稳定"，建议重新评估需求 |
| 变更涉及P0需求 | 必须产品负责人+技术负责人双签 |
| PRD新旧版本均缺失 | 基于变更描述记录，标注"缺乏PRD对比" |

## 质量检查

- [ ] 变更有唯一编号
- [ ] 变更前后描述清晰
- [ ] 影响评估覆盖5个维度
- [ ] 受影响需求和文档已列出
- [ ] 决策建议有依据
- [ ] 审批流程已定义
- [ ] 变更日志汇总完整

## 降级策略

| 缺失的上游输入 | 降级方案 | 输出影响 |
|---------------|---------|---------|
| PRD旧版本缺失 | 变更前描述基于用户提供，标注"缺乏PRD原文" | 变更差异可能不够精确 |
| PRD新版本缺失 | 变更后描述基于用户提供，标注"缺乏PRD原文" | 变更差异可能不够精确 |
| SRS缺失 | 受影响文档不包含SRS | 下游影响评估可能不完整 |
| 迭代信息缺失 | 不关联具体迭代 | 无法统计迭代变更率 |
